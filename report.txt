EXECUTIVE SUMMARY
================================================================================

DevCred AI is a full-stack web application that validates resume claims against
actual GitHub repositories using artificial intelligence. The application employs
a modern tech stack (React frontend, FastAPI backend) and integrates with Google
Gemini AI for intelligent resume analysis combined with live GitHub API data
for real-time project verification.

The system is designed to detect false claims, identify tutorial clones, and
provide confidence-based verification without data congestion or false positives.

SECTION 1: END-TO-END USER WORKFLOW
================================================================================

1.1 RESUME UPLOAD PHASE
------------------------

Step 1: User visits DevCred AI dashboard (React frontend, localhost:5173)
Step 2: User selects resume file (PDF or DOCX format)
Step 3: User optionally enters GitHub username for validation
Step 4: User clicks "Analyze Resume" button
Step 5: Frontend creates FormData object containing:
        - File: Resume document (binary)
Step 6: Frontend sends POST request to backend:
        - Endpoint: http://127.0.0.1:8000/api/v1/resume/parse
        - Query param: ?github_username=username (if provided)
        - Body: multipart/form-data with file
Step 7: Request intercepted by FastAPI router in main.py

1.2 BACKEND FILE PARSING PHASE
--------------------------------

Step 1: FastAPI route handler validates file:
        - Checks filename exists
        - Extracts file extension
        - Validates format (PDF or DOCX only)
        - Checks file is not empty

Step 2: File content read into bytes:
        - file_content = await file.read()

Step 3: Route calls parse_resume() from parsers.py module:
        - Passes: file_content (bytes), file_extension (str)
        - Returns: extracted_text (string)

Step 4a: PDF PARSING (if .pdf)
         - Uses pypdf library
         - Initializes PdfReader from BytesIO stream
         - Iterates through all PDF pages
         - Calls extract_text() on each page object
         - Joins pages with \n\n separator for structure

Step 4b: DOCX PARSING (if .docx)
         - Uses python-docx library
         - Initializes Document from BytesIO stream
         - Iterates through all paragraph objects
         - Extracts text and filters empty paragraphs
         - Joins paragraphs with \n separator

Step 5: Returns to route with clean, extracted text

1.3 GITHUB INTEGRATION PHASE (Optional)
-----------------------------------------

Step 1: Route checks if github_username provided:
        if github_username: fetch_repositories = True

Step 2: If username provided, call github_service.fetch_user_repositories():
        - Validates username is not empty
        - Constructs GitHub API URL:
          https://api.github.com/users/{username}/repos
        - Parameters: type=public, sort=updated, per_page=100
        - Uses httpx.AsyncClient for async HTTP request
        - Timeout: 10 seconds

Step 3: GitHub API Response handling:
        - 404: User not found → raises GitHubServiceError
        - 403: Rate limit exceeded → raises GitHubServiceError
        - 200: Success → parse JSON response
        
Step 4: Transform API response to metadata schema:
        For each repository in response:
        {
          "name": repo.name,
          "description": repo.description (or null),
          "primary_language": repo.language (or null)
        }

Step 5: Returns list of metadata dictionaries to route

Step 6: If no username provided:
        github_metadata = [] (empty list)

1.4 GEMINI AI ANALYSIS PHASE
------------------------------

Step 1: Route calls extract_resume_analysis() from ai_service.py:
        - Passes: resume_text (string), github_metadata (list)

Step 2: Function builds structured output config:
        - response_mime_type: "application/json"
        - response_schema: ResumeAnalysis (Pydantic model)
        - temperature: 0.0 (deterministic output)

Step 3: Function constructs system prompt (RECRUITER_SYSTEM_PROMPT):
        "You are an expert technical recruiter..."
        Instructions to extract:
        - Candidate name
        - Technical skills (explicit mentions only)
        - Project descriptions and validation

Step 4: Function builds GitHub context if metadata provided:
        - Format each repo as string
        - Include: name, description, language
        - Append to system prompt as reference data

Step 5: Function calls Gemini API:
        - Model: gemini-3.1-flash-lite
        - Method: client.models.generate_content()
        - Inputs: model, prompt text, config
        - Temperature: 0.0 for deterministic output

Step 6: Gemini AI Processing:
        a) Reads resume text completely
        b) Identifies candidate name (first section, headers, etc.)
        c) Extracts all explicitly mentioned technical skills
        d) Isolates project descriptions
        e) Compares projects with GitHub repository list
        f) Cross-references by:
           - Project name similarity
           - Description correlation
           - Technology/language alignment
        g) Assigns verification status:
           - "Verified" if resume project correlates with actual repo
           - "Tutorial Clone" if repo shows boilerplate patterns
           - "Unverified" if no match found
        h) Provides confidence_reasoning (why it chose that status)

Step 7: Response parsing:
        - Gemini returns JSON string
        - Function parses JSON to Python dict
        - Validates against ResumeAnalysis Pydantic schema
        - If invalid: raises AIServiceError
        - If valid: continues

Step 8: Function constructs response dictionary:
        {
          "candidate_name": extracted_name,
          "detected_skills": [skill1, skill2, ...],
          "extracted_projects": [
            {
              "project_name": "...",
              "resume_description": "...",
              "matching_github_repo": "repo_name or null",
              "verification_status": "Verified|Tutorial Clone|Unverified",
              "confidence_reasoning": "..."
            },
            ...
          ]
        }

Step 9: Returns to route

1.5 RESPONSE ASSEMBLY & RETURN PHASE
-------------------------------------

Step 1: Route assembles final response object:
        {
          "status": "success",
          "filename": original_filename,
          "file_type": "PDF" or "DOCX",
          "github_validation": {
            "status": "success" or "skipped",
            "username": provided_username or null,
            "repositories_found": count of repos
          },
          "raw_text": extracted_text,
          "analysis": {
            "candidate_name": ...,
            "detected_skills": [...],
            "extracted_projects": [...]
          }
        }

Step 2: Route returns JSON response with HTTP 200 status

Step 3: Frontend receives response:
        - Response.json() parses JSON payload
        - Sets analysisResult state

Step 4: React re-renders component:
        - Displays candidate name in large heading
        - Shows skill pills with modern styling
        - Lists projects with verification status badges
        - Displays GitHub validation summary
        - Color-codes: Verified (green), Tutorial Clone (amber), Unverified (gray)

Step 5: User sees complete analysis on dashboard

SECTION 3: DATA FLOW ARCHITECTURE
================================================================================

Request Journey (Client to Server):

1. User in browser → React App (App.tsx)
2. App.tsx → fetch() → POST /api/v1/resume/parse
3. Network → HTTP request to FastAPI backend
4. FastAPI main.py (route handler) receives request
5. route handler → parsers.py (parse file)
6. route handler → github_service.py (fetch repos if username)
7. route handler → ai_service.py (analyze with Gemini)
8. ai_service.py → google.genai API (external Gemini)
9. ai_service.py ← Response from Gemini
10. route handler ← Analysis result
11. FastAPI → HTTP response (JSON) to browser
12. React App ← fetch() resolves with response.json()
13. App.tsx → setState(analysisResult)
14. React re-renders with results

Data Transformation:

Resume File (binary)
  ↓
parsers.py → extract_text()
  ↓
Resume Text (string)
  ↓
+ GitHub Username (optional)
  ↓
github_service.py → fetch_repos()
  ↓
Repository Metadata (list of dicts)
  ↓
+ Extracted Text + Repo Metadata
  ↓
ai_service.py → Gemini API
  ↓
Gemini Analysis (JSON)
  ↓
Pydantic Validation
  ↓
Structured Analysis (typed dict)
  ↓
HTTP JSON Response
  ↓
Frontend Display

SECTION 4: DESIGN DECISIONS & TRADE-OFFS
================================================================================

Choice 1: Async/Await Pattern
Decision: Use async functions throughout (async def, await)
Rationale:
  - Non-blocking I/O for file operations
  - Scalable for concurrent requests
  - Compatible with FastAPI's async capabilities
Trade-off: Requires understanding of async Python

Choice 2: Structured Output with Pydantic
Decision: Use Pydantic schemas for AI response validation
Rationale:
  - Type safety at runtime
  - Automatic validation of AI output
  - Clear contract for API responses
Trade-off: Adds validation overhead but improves reliability

Choice 3: In-Memory Cache (Mocked Redis)
Decision: Use simple dict for caching instead of real Redis
Rationale:
  - Eliminates Redis dependency for development
  - Faster local development loop
  - Easy to switch to real Redis in production
Trade-off: No persistence across restarts, no distributed caching

Choice 4: SQLite for Development
Decision: Use SQLite as default database
Rationale:
  - Zero-configuration
  - File-based (easy to test)
  - No external service needed
Trade-off: Not suitable for production; must migrate to PostgreSQL

Choice 5: Tailwind CSS v4 with @import
Decision: Single @import "tailwindcss" instead of v3 @tailwind directives
Rationale:
  - Latest Tailwind CSS best practices
  - Unified directive system
  - Better CSS output optimization
Trade-off: Requires newer build tools (Vite, PostCSS)

Choice 6: Public GitHub API Only
Decision: Only fetch public repositories, no private access
Rationale:
  - User privacy protection
  - No need for elevated GitHub permissions
  - Safer, simpler authentication
Trade-off: Cannot validate private projects

Choice 7: Gemini AI for Analysis
Decision: Use Google Gemini API instead of alternatives
Rationale:
  - Free tier available for development
  - Structured output support
  - Good accuracy for technical extraction
Trade-off: API quota limits, external dependency
